# PI Assistant Composer Inputs, File Support, and Run Analytics Plan

## Goal
Build one shared architecture for native image pasting, run-scoped outcome capture/local analytics, and future file/input support.

The key idea is simple: **composer inputs, canonical transcript content, and analytics are related, but they are not the same thing**. They need different lifecycles, different storage, and different sources of truth.

## Core architectural decision
Treat the system as three distinct state domains:

| State domain | Owner | Retention | Source of truth |
|---|---|---|---|
| Pending composer input state | Host | Until remove, send success, close, or restart | Host store |
| Canonical conversation content | PI session history | Session lifetime | PI session JSONL |
| Local run analytics | Host analytics service + local storage | Until export/purge | Local analytics store |

Why this matters:
- A pasted image before send is **not** canonical conversation history.
- A sent image **is** canonical conversation history and must come back on reopen.
- A user outcome score is **not** conversation history; it is local analysis data.
- Analytics should observe the structured send envelope **before** it is lowered into transport-specific behavior.

## Product principles
1. **Host-owned session state for non-text inputs.** Pending non-text inputs must survive webview teardown, session switches, and send failure.
2. **Canonical transcript data comes from PI session history.** Reopen correctness must not depend on temp files or webview memory.
3. **Runs, not sessions, are the analytics unit.** A long-lived tab can contain multiple unrelated tasks.
4. **Local-first, append-only analytics.** Store structured facts locally; avoid remote sync and noisy raw telemetry by default.
5. **Do not collapse distinct input types into text too early.** Once structure is flattened, transcript fidelity and analytics quality both suffer.
6. **Do not invent an attachment file store unless there is no better option.** Native PI content should remain in canonical session history.
7. **Future file support must distinguish references from payloads.** A filesystem path and a pasted binary blob are not the same kind of input.

## Shared domain model
### Session
The persisted PI conversation/tab.

### Run
One analyzable unit of work inside a session. A run starts on the first user send for that task and ends when the user scores it or it closes unscored.

### Outcome
An explicit user judgment for a run: resolution, satisfaction rating, and optional reason tags later.

### Composer input
A pending input attached to a session before send.

```ts
type ComposerInput =
  | {
      id: string;          // host-assigned UUID on add; host is authoritative owner
      kind: 'filesystemPathRef';
      path: string;
      name: string;
      source: 'picker' | 'drop';
    }
  | {
      id: string;
      kind: 'imageBlob';
      mimeType: string;
      name: string;
      sizeBytes: number;
      dataBase64: string;
      width?: number;
      height?: number;
      source: 'paste' | 'drop';
    }
  | {
      id: string;
      kind: 'fileBlob';   // reserved — rejected by the backend until Phase 6
      mimeType: string;
      name: string;
      sizeBytes: number;
      dataBase64: string;
      source: 'paste' | 'drop';
    };
```

Important distinctions:
- `filesystemPathRef` is a reference to an existing filesystem path.
- `imageBlob` is native image content.
- `fileBlob` is present in the type union so the protocol and UI layers can recognize and handle it gracefully, but the backend **must reject it** with a user-facing error until Phase 6. Including it now avoids silent type-checking failures in the webview if a user somehow triggers a non-image binary paste before that phase ships. Do not implement it as hidden temp-file attachment emulation.

`id` is generated by the **host** when it receives an `addComposerInput` message. The webview sends raw input data; the host assigns a stable ID and owns the canonical keyed map. This prevents ID collisions across sessions and decouples the webview's clipboard / drop event handling from identity concerns.

### Canonical chat content
The transcript model must become rich enough to preserve user image content. The current `ChatMessagePart` type covers only assistant message structure (text, reasoning, tool calls). User messages need a separate type to hold image parts.

```ts
// New type — used in ChatMessage.userParts for role === 'user' messages.
type UserContentPart =
  | { kind: 'text'; text: string }
  | {
      kind: 'image';
      mimeType: string;
      dataBase64: string;
      name?: string;
      width?: number;
      height?: number;
    };

// Existing type — kept unchanged for role === 'assistant' messages.
type AssistantContentPart =   // rename of ChatMessagePart for clarity in Phase 1
  | { kind: 'text'; text: string }
  | { kind: 'reasoning'; text: string }
  | { kind: 'toolCall'; toolCall: ToolCall };
```

**`ChatMessage` migration strategy:** Add a new optional `userParts?: UserContentPart[]` field to `ChatMessage` for user messages that have image content. The existing `parts?: ChatMessagePart[]` field remains for assistant messages and is left unchanged. The existing `markdown` field stays as a derived compatibility field for both roles during the migration window.

This avoids a discriminated union on `ChatMessage.role` and does not require changes to the assistant streaming pipeline. Consumers rendering user messages check `userParts` if present; if absent, they fall back to `markdown` as before.

Short-term, `markdown` can remain as a derived compatibility field for both roles. For user messages with image content, `markdown` should contain the text portion only (not the image), so that text-only consumers degrade gracefully.

### Model capability metadata
The UI needs explicit capability data.

```ts
interface ModelInfo {
  id: string;
  name: string;
  provider: string;
  reasoning: boolean;
  inputKinds: Array<'text' | 'image'>;  // new field
  contextWindow?: number;
  maxTokens?: number;
}
```

**SDK mapping:** The backend populates `inputKinds` from the SDK model registry. If the registry does not expose input capability information for a given model, the backend must default to `['text']` only — never assume image capability. An empty or absent registry field must not be treated as implicit image support.

**Model switch with pending image inputs:** When the active model is changed and the session has pending `imageBlob` entries in its composer inputs, the host must not silently discard those images or silently block the switch. Required behaviour: surface a notice informing the user that the newly selected model does not support image inputs and that pending images will be removed if they proceed, then clear only the `imageBlob` entries from the session's pending input set on confirmation. `filesystemPathRef` inputs are unaffected by model capability and must not be cleared. This guard lives in the host (`extension-host.ts` `setModel` handler), where both the pending input set and the incoming model's `inputKinds` are known simultaneously.

### State contract alignment
Adding `pendingComposerInputsBySession` to host state extends the existing session-scoped state pattern. `docs/STATE_CONTRACT.md` must be updated in Phase 1 to include pending composer inputs in the list of session state that is cleared on session close and extension restart. This is a documentation requirement before implementation, not after.

## The critical distinction: references vs native media vs future artifacts
This boundary is what lets both plans coexist cleanly.

| Input kind | Pending owner | Send behavior | Canonical after send | Analytics meaning |
|---|---|---|---|---|
| Filesystem path reference | Host | Lower to prompt text or future native file-ref path | Usually just text today | "used file references" |
| Pasted image | Host | Send as native PI image input | Native image content in session history | "used image input" |
| Arbitrary pasted file blob | Host | Rejected with user-facing error until Phase 6 | None | "unsupported input attempted" |

Consequences:
- The current `pendingPaths -> @path` behavior is acceptable for **filesystem path references**, but it is the wrong architecture for **images**.
- Images should not be implemented as temp files.
- Analytics must observe structured composer inputs **before** send lowering.

**Filesystem path reference long-term:** The current `@path` lowering is explicitly short-term. It becomes a candidate for native path-reference support if and when the PI SDK exposes one. This should be revisited at the start of Phase 6, not left open-ended.

## Lifecycle and retention rules
### Pending composer inputs
Pending inputs should be host-owned, keyed by `sessionPath`, visible in `ViewState`, retained across webview reload/session switching/send failure, and deleted on remove, send success, session close, or extension restart/shutdown.

This extends the host-owned state pattern already present in the extension and aligns with `docs/STATE_CONTRACT.md`.

**Image size limit:** The host must enforce a per-image size limit before storing a pending `imageBlob`. A reasonable default is 10 MB (`sizeBytes` ≤ 10 × 1024 × 1024). Inputs that exceed this limit are rejected at the `addComposerInput` handler with a user-facing notice. The validation in `rpc.ts` must enforce the same limit before forwarding to the backend.

**ViewState snapshot size:** `pendingComposerInputs` in `ViewState` includes Base64 image data, which makes full snapshots larger. This is acceptable for typical use (one or two pasted images at a time). The 10 MB per-image limit above bounds the worst-case size. If snapshot transfer becomes a bottleneck in the future, pending image data could be stripped from the snapshot and delivered via a separate channel, but this is not required in Phase 1 or 2.

### Committed conversation content
Committed content should always be rendered from canonical session history:
- text remains text
- sent images should be persisted by PI and restored from transcript mapping
- no extension-managed temp files for committed images
- transcript display should prefer canonical session data over leftover pending state

**PI SDK persistence assumption:** Acceptance criterion #4 (reopen restores user images) depends on the PI SDK persisting image content in the session JSONL when `session.prompt(text, { images })` is called. This must be verified during Phase 2 implementation before shipping.

**Fallback if SDK does not persist image bytes:** If verification shows the SDK does not write image bytes into the session JSONL, the extension must implement its own image content store rather than leaving criterion #4 unmet. Concrete design: store image bytes under `globalStorageUri/image-store/<workspaceHash>/`, keyed by the `sha256` hash of the image content (filename `<hash>.<ext>`). Written once on send success; never mutated. `ChatMessage.userParts` image entries carry a `storeKey` field referencing the hash instead of inline `dataBase64`; the transcript mapper resolves `storeKey` references to bytes at render time. Reachability is determined at startup by scanning open session JSONLs for referenced keys; unreferenced entries are eligible for compaction. This is the one scenario where an extension-managed image store is justified — it is content-addressed and reachability-tracked, not a temp-file hack. If this path is taken, acceptance criterion #11 must be revised to explicitly carve out content-addressed committed image storage from the "no separate attachment cleanup system" requirement.

### Analytics data
Analytics should be local-first, append-only where practical, schema-versioned, and limited to structured counts/flags/hashes by default. Raw transcript or tool payload storage should be opt-in later, not part of the baseline architecture.

## Recommended protocol architecture
The current protocol is too text-centric. It should become input-aware and run-aware.

### Protocol version
Any addition of new message types or breaking changes to existing shapes requires a `PROTOCOL_VERSION` bump in `protocol.ts`. The changes in this plan (new message types, modified `sendRejected`, new `ViewState` fields) constitute at minimum one protocol bump when Phase 1 ships. The `extension/src/host/extension-host.ts` version check must be updated accordingly.

### Host-owned composer input flow
Preferred direction:
- webview -> host: `addComposerInput` (raw input data, no ID)
- host: assigns ID, stores input keyed by session path
- host -> webview: active-session snapshot includes pending composer inputs (with IDs)
- webview -> host: `removeComposerInput` (by ID)
- webview -> host: `send` carries text only (no inputs, no pendingPaths)
- host -> backend: send uses the host-owned input set for that session

This is better than making the webview the source of truth for pending inputs because it reduces race conditions, simplifies failure restore, makes session switching/reload work naturally, and lets analytics see the authoritative input set.

### Send payload (webview → host)
The existing `{ type: 'send'; text: string; pendingPaths?: string[] }` message in `WebviewToHostMessage` must be updated. `pendingPaths` is removed; inputs are now entirely host-owned. The updated shape is:

```ts
{ type: 'send'; text: string }
```

Empty text is valid when the host-owned input set for the session is non-empty (image-only send). The host is responsible for validating that at least one of text or pending inputs is non-empty before forwarding to the backend.

### `sendRejected` message (host → webview)
`pendingPaths` is removed from `sendRejected` once host-owned inputs are in place. The updated shape is:

```ts
{ type: 'sendRejected'; sessionPath: string; text: string }
```

The webview only restores the text draft on rejection. Pending inputs are already in host state and remain there on failure without the webview needing to act.

### Send payload (host → backend)
The backend contract should be explicit:

```ts
interface MessageSendParams {
  sessionPath: string;
  text: string;   // may be empty string if inputs are non-empty
  inputs?: ComposerInput[];
}
```

`rpc.ts:validateMessageSend` must be updated to allow empty text **when `inputs` is non-empty**. The combined invariant is: at least one of `text.trim()` or `inputs.length > 0` must be true.

### View state additions
`ViewState` should include active-session pending inputs and basic run state.

```ts
interface ViewState {
  // existing fields...
  pendingComposerInputs: ComposerInput[];
  activeRunSummary: {
    runId: string;
    status: 'open' | 'scored' | 'closed_unscored';
    scored: boolean;
    // taskGroupId is intentionally omitted — the webview has no rendering use for it.
    // Context menu actions are driven by status alone.
  } | null;
}
```

`activeRunSummary` uses explicit `null` (not optional) so consumers can reliably distinguish "no active run" from "field not yet populated."

### Draft restore
Once pending inputs are host-owned, `sendRejected` no longer needs to restore attachment state. It only needs to restore the text draft.

## Backend lowering strategy
The backend should receive structured inputs and lower them deliberately.

### Filesystem path references
Short-term behavior:
- lower to the existing `@path` text prelude before calling `session.prompt`
- treat this as prompt construction, not canonical attachment persistence

This preserves current functionality without pretending that file references are native transcript media.

### Images
Required behavior:
- map `imageBlob` inputs to PI SDK image content
- call `session.prompt(text, { source: 'rpc', images })`
- preserve image content when mapping session history back into transcript messages

This is the correct end-to-end path for images.

### Future arbitrary file blobs
Do **not** ship arbitrary pasted file blobs through temp files as a first response. The backend must reject `fileBlob` inputs with a clear error message until Phase 6. Only support them when one of these becomes true:
1. PI exposes a native file-input path with canonical persistence, or
2. the extension introduces a real artifact store

If an artifact store is ever needed, it should be content-addressed, reference-counted or reachability-tracked, clearly modeled as an artifact system, and separate from temp-file hacks.

## Transcript architecture
This is where the two plans meet reality.

Current limitation: the extension already preserves rich assistant structure, but user messages are still effectively flattened to markdown. That blocks restored user images, accurate optimistic local user rows with image content, and future analytics that differentiate text-only from image-assisted runs.

Recommended direction:
1. Add `userParts?: UserContentPart[]` to `ChatMessage` for user messages with image content.
2. Keep `markdown` as a derived compatibility field for both user and assistant messages.
3. Render user image parts in the transcript by checking `userParts` first, falling back to `markdown`.
4. Update restored transcript mapping so reopened sessions reconstruct `userParts` from JSONL image content (conditional on PI SDK verification — see acceptance criterion #4).
5. Keep filesystem path references as prompt text unless/until PI supports native file references.

### Optimistic local messages
When the user sends a prompt:
- optimistic local user rows should be built from the structured send envelope, including `userParts` with any pending image inputs
- images can be shown immediately in the optimistic row
- on failure, the optimistic row is removed and pending inputs remain host-owned (no special webview recovery needed)
- on success, canonical transcript data replaces the optimistic state; if the session is busy when `session.opened` arrives, the local optimistic row is preserved as per the existing transcript resolution logic

`transcriptActions.appendLocalUserMessage` must be updated (or a new action added) to accept `userParts?: UserContentPart[]` in addition to `text`.

## Run analytics architecture
### Unit of analysis
The analytics unit must be a **run**, not a session. A run should contain one task attempt, one treatment/config snapshot at start, one explicit outcome if scored, and derived aggregates from host/backend events. A session may contain multiple runs.

### Run lifecycle
1. Opening or creating a tab does not create a run.
2. The first user send in a task starts a run.
3. `Record outcome` finalizes the current run.
4. `Start new task` marks the intent to start a new `task_group_id` on the next send. If a run is currently open (agent mid-response), the action queues — the task group boundary is recorded at the next send, not during streaming. The open run is finalized as `closed_unscored` if the user never scores it before sending again.
5. `Continue / retry same task` preserves the task group; the next send opens a new run within the same group.
6. Closing a tab with an open run finalizes it as `closed_unscored`.
7. Config/treatment changes mid-run mark it `mixed` for experimentation.

### StatsService event interface
`StatsService` observes `SessionService` through an explicit callback interface, not by reading Redux state or scraping the transcript. `SessionService` receives a `RunObserver` at construction time. This keeps `StatsService` decoupled and testable.

```ts
interface RunObserver {
  onSend(sessionPath: string, runId: string, inputs: ComposerInput[]): void;
  onRunStarted(sessionPath: string, runId: string, taskGroupId: string): void;
  onAssistantTurnStarted(sessionPath: string, runId: string, requestId: string): void;
  onAssistantTurnEnded(sessionPath: string, runId: string, requestId: string, durationMs: number): void;
  onInterrupted(sessionPath: string, runId: string): void;
  onMessageEdited(sessionPath: string, runId: string, messageId: string): void;
  onTruncatedAfter(sessionPath: string, runId: string, messageId: string): void;
  onBackendError(sessionPath: string, runId: string | null, code: string): void;
  onContextUsageChanged(sessionPath: string, tokens: number | null, limit: number): void;
  onRunFinalized(sessionPath: string, runId: string, reason: 'scored' | 'closed_unscored' | 'new_task'): void;
}
```

`SessionService` calls observer methods at the appropriate host/backend event boundaries. If `StatsService` is not yet constructed (Phase 1–2), a no-op stub satisfies the interface.

`onMessageEdited` and `onTruncatedAfter` are among the highest-signal quality events in the system — they indicate the user found a prior response wrong or off-track. Both are included in the interface from Phase 1 so the contract is stable across all phases; their `StatsService` implementations are deferred to Phase 5. No interface changes are required at Phase 5.

### What the collector should observe
The collector should live in the host and subscribe to authoritative host/backend boundaries through the `RunObserver` interface, not scrape rendered UI. High-value facts include:
- run start/end
- user sends
- composer input summary at send time
- assistant turn count and durations
- busy time
- tool calls and failures
- subagent usage
- message edits
- reruns / truncate-after flows
- interruptions
- backend errors
- context usage
- model / thinking settings
- prompt/skill/tool metadata once exposed structurally
- explicit user outcome

### Phase 2 → Phase 3 compatibility
Phase 2 ships image sends before Phase 3 ships `StatsService`. The `RunObserver` stub ensures send events during Phase 2 are no-ops analytically. Phase 3 adds a real implementation. No structural changes to `SessionService` are needed between the phases because the `RunObserver` interface is established in Phase 1. By the same logic, Phase 5 (`onMessageEdited`, `onTruncatedAfter` implementation) also requires no interface changes — the methods are already in the contract.

### Input/file metrics to capture
Because this plan unifies file support with analytics, the collector should explicitly record:
- `filesystemPathRefCount`
- `imageInputCount`
- `imageInputBytes`
- `unsupportedInputCount`
- `inputKindsUsed`

This matters because if everything is flattened to text first, later analysis cannot answer simple questions like:
- do image-assisted runs score differently?
- do file-reference-heavy runs perform better on debugging tasks?
- do unsupported-input errors correlate with dissatisfaction?

## Persistence strategy for analytics
Use a storage abstraction from the start.

Recommended Phase 1 storage:
- append-only `run-snapshots.jsonl`
- append-only `outcome-history.jsonl`
- alternating-slot `open-runs.a.json` / `open-runs.b.json` checkpoint pair with `open-runs.gen` slot indicator

**Storage location:** All analytics files are stored under VS Code's `globalStorageUri` for the extension, organized by hashed workspace identifier (e.g., `runs/<workspaceHash>/`). Global storage ensures data is not lost when a workspace folder is removed, while the workspace hash keeps per-project runs logically separated.

**Crash consistency:** A write-to-`.tmp`-then-rename strategy is not safe on Windows: `fs.rename` over an existing file throws `EPERM`. Use a two-alternating-slots strategy instead: maintain `open-runs.a.json` and `open-runs.b.json` alongside an `open-runs.gen` file that records the currently-active slot (`a` or `b`). To checkpoint: write the new state to the *inactive* slot, then overwrite `open-runs.gen` with the new slot name. A crash during the inactive-slot write leaves the active slot intact. A crash during the `open-runs.gen` write is recoverable: on startup, read `open-runs.gen` to find the active slot; if it is absent or unparseable, read both slots and prefer the one with the higher `seq` field. This strategy never renames over an existing file and is correct on all platforms. JSONL files are append-only and safe for partial-write tails; on read, the last line is validated and discarded if it does not parse.

That is enough to start collecting useful labeled data without blocking on SQLite packaging. Later, a query-friendly DB view can sit behind the same interface.

## Privacy and data minimization
Default analytics posture should be:
- local only
- no automatic sync
- workspace identifiers hashed
- prompt/config text stored as hashes/IDs by default
- no raw transcript or raw tool payload persistence by default
- store structured counts and flags, not streaming deltas

If excerpts are stored later, they should be explicit and limited.

## Recommended services and ownership
### `SessionService`
Keeps ownership of backend lifecycle, session operations, optimistic transcript state, and session-scoped pending composer inputs for send/retry flows. Notifies the `RunObserver` at event boundaries.

### `StatsService` (new)
A new host service should own open run state, run lifecycle transitions, outcome recording, snapshot finalization, persistence of scored and unscored runs, and crash/shutdown checkpointing. Implements `RunObserver`.

### Backend
Continues to own lowering structured send inputs into PI SDK calls, transcript reconstruction from session history, model capability exposure, and structured prompt/tool/skill metadata emission where needed.

### Webview
Should stay presentation-focused: paste handling, file picking UI, composer previews/chips, transcript rendering, tab context menu for outcome actions, and sending typed intent to the host. It should not become the source of truth for pending inputs or run state.

## Phased rollout
### Phase 1 — Shared foundation
- Update `STATE_CONTRACT.md`.
- Bump `PROTOCOL_VERSION`.
- Implement `ComposerInput`, `UserContentPart`, `ModelInfo.inputKinds`.
- Host-owned pending inputs with add/remove protocol messages.
- Update `sendRejected` and `send` message shapes.
- Update `ViewState` with `pendingComposerInputs` and `activeRunSummary`.
- Establish `RunObserver` interface with no-op stub.
- Keep filesystem path references working through the current `@path` lowering path.

### Phase 2 — Native image support
- Paste/drop image into composer; preview chip with remove.
- Native backend image send path.
- `validateMessageSend` update for empty-text image-only sends.
- Optimistic local user row with image content.
- Restored transcript rendering for user images (conditional on PI SDK JSONL persistence — verify before shipping).
- Send-failure recovery via host-owned pending inputs.

### Phase 3 — Run identity and outcome capture
- Implement `StatsService` with `RunObserver` subscription.
- Run/task-group lineage.
- **Model and thinking-level snapshot at run start.** Captured here, not Phase 4, because a run snapshot without its treatment config is not useful for model-comparison analysis. Phase 4 run data would be incomparable by model if this is deferred.
- Tab context-menu actions.
- Explicit outcome capture.
- Scored/unscored persistence under `globalStorageUri`.

### Phase 4 — Structured factor capture
- Prompt family and hash capture.
- Skill/tool metadata capture.
- Input/file usage metrics in run snapshots.
- Treatment purity tracking.

### Phase 5 — Verification, mutation, and analysis improvements
- Implement `StatsService` handlers for `onMessageEdited` and `onTruncatedAfter` (interface and no-op stubs established in Phase 1; no `SessionService` or interface changes required).
- File-mutation rollups.
- Verification command classification.
- Export/query paths.
- Experiment-assignment plumbing.

### Phase 6 — Future arbitrary file payload support and filesystem path reference revisit
- Implement arbitrary pasted file blobs only if justified by native SDK file inputs or a real artifact store.
- Revisit filesystem path reference lowering: if the SDK now exposes a native file-reference path, migrate `filesystemPathRef` handling away from `@path` text prelude.
- Do not implement either as disguised temp-file attachment emulation.

## Acceptance criteria
This consolidated plan is implemented correctly when all of the following are true:
1. Pasting an image creates a visible host-owned pending input.
2. Pending inputs survive session switches, webview reloads, and send failure.
3. Sending an image uses PI native image input, not temp-file emulation.
4. Reopening a session restores user images from canonical session history. *(Phase 2 pre-ship gate: verify PI SDK persists image content in JSONL on `session.prompt` with images.)*
5. Filesystem path references still work, but remain clearly modeled as references rather than canonical media attachments.
6. The first user send starts a run; opening a tab does not.
7. Users can record outcomes and task lineage from session tabs.
8. Unscored runs are persisted with an explicit reason.
9. Analytics snapshots capture structured input/file usage counts without storing raw noisy telemetry by default.
10. Models without image support are gated with a clear user-facing message; image inputs are disabled in the composer for those models.
11. No separate attachment cleanup system exists for committed images.
12. Future arbitrary file payload support remains blocked on native support or a proper artifact store; `fileBlob` inputs return a user-facing error from the backend in all phases before Phase 6.
13. `sendRejected` restores text draft only; no `pendingPaths` field exists.
14. `PROTOCOL_VERSION` is bumped before any wire-protocol changes ship.

## Main risks and constraints
1. **Transcript model migration** is the biggest structural change because the current model is assistant-rich but user-poor.
2. **PI SDK image persistence** is an unverified assumption for criterion #4. If the SDK does not persist images in JSONL, the content-addressed image store fallback described in the committed conversation content section must be implemented before Phase 2 ships transcript restore. The fallback design is fully specified; it is not a blocker, but it is additional Phase 2 scope if needed.
3. **Base64 payload size and memory pressure** are bounded by the 10 MB per-image limit. Enforce at `addComposerInput` and `validateMessageSend`.
4. **Session file growth** is expected once images are persisted natively.
5. **Analytics privacy creep** must be actively resisted.
6. **Treatment drift in experiments** requires `mixed`-run handling.
7. **Future file support temptation** should not lead back to temp-file hacks.
8. **Analytics crash safety** depends on correct checkpoint writes; use the two-alternating-slots strategy (`open-runs.a/b.json` + `open-runs.gen`) — not write-then-rename, which is not safe on Windows.

## Final recommendation
Use this combined architecture as the canonical plan.

The most important decisions are:
- host-own pending composer inputs
- treat runs, not sessions, as the analytics unit
- preserve canonical media in PI session history
- observe structured inputs before lowering them into transport details
- distinguish file references from binary payloads
- defer arbitrary binary file support until native support or a real artifact store exists
- verify PI SDK image persistence before shipping Phase 2 transcript restore

If those decisions are made now, image pasting, outcome analytics, future file support, and future experiments can all grow from the same foundation instead of fighting each other later.
